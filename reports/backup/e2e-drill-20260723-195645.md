## Design constraint: auth-DB circular dependency

> 注：此 auth-DB 循环依赖已于 Round 6 / F3 通过 `--restore` CLI 解决（v1.2.0）。当前推荐的灾备恢复路径为 `docker exec contentx /app/contentx --restore <file>`，详见 SOP §3.4。Scenario B 的 psql 直连为历史 workaround。

The restore endpoint POST /api/v1/admin/backup/:file/restore requires admin authentication. The auth middleware (internal/middleware/auth.go) queries the users table (with Preload Role and Preload Role.Permissions) on every request. If the entire database is lost (DROP SCHEMA), the users table is gone and the endpoint returns 401 User not found -- so the restore endpoint **cannot** be used for total-DB-loss recovery.

Two scenarios cover both the endpoint proof and the realistic disaster path:
- **Scenario A** keeps auth tables intact (TRUNCATE business tables only) so the admin token still validates, then calls the restore endpoint. This proves the endpoint works.
- **Scenario B** drops the entire schema (total loss) and restores via direct psql -f from the app container. This is the realistic disaster-recovery path operators must use.

**Note on activity_logs**: the ctivity_logs table is excluded from strict row-count comparison. The ActivityLogger middleware appends a row for every API call, so the drill itself (backup, restore, functional checks) naturally grows this table. It is operational metadata, not business data; a small positive delta is expected and not a restore regression.

## Reproduction Commands

```powershell
# 0. Start stack (app image must include postgresql-client)
docker compose up -d

# Scenario A: API restore over working DB
# 1. Login
$resp = Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:18080/api/v1/auth/login' 
  -ContentType 'application/json' -Body '{"username":"admin","password":"<ADMIN_PASSWORD>"}'
$token = $resp.data.token.access_token

# 2. Backup
$bk = Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:18080/api/v1/admin/backup?type=db' 
  -Headers @{ Authorization = 'Bearer ' + $token }
$bkFile = $bk.data.path

# 3. Truncate business tables (keep auth tables)
docker exec contentx-db psql -U contentx -d contentx -c "DO $$ DECLARE t text; BEGIN FOR t IN SELECT tablename FROM pg_tables WHERE schemaname='public' AND tablename NOT IN ('users','roles','permissions','role_permissions','schema_migrations') LOOP EXECUTE 'TRUNCATE TABLE public.' || quote_ident(t) || ' CASCADE'; END LOOP; END $$;"

# 4. Restore via API
Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:18080/api/v1/admin/backup/db-20260723-115646.sql/restore' 
  -Headers @{ Authorization = 'Bearer ' + $token }

# Scenario B: Total-loss recovery via direct psql
# 5. Drop entire schema
docker exec contentx-db psql -U contentx -d contentx -c 'DROP SCHEMA public CASCADE; CREATE SCHEMA public;'

# 6. Restore via direct psql from app container (auth endpoint is unusable)
docker exec contentx sh -c 'PGPASSWORD=$DB_PASSWORD psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f /app/backups/db-20260723-115646.sql'

# 7. Verify
docker exec contentx-db psql -U contentx -d contentx -c 'SELECT count(*) FROM articles;'
```
